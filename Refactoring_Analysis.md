# Анализ проекта KosmosCore на необходимость рефакторинга

В данном отчете представлен подробный анализ текущего состояния кодовой базы проекта KosmosCore. Выявлены ключевые проблемы в архитектуре, безопасности, производительности и организации кода, а также предложены конкретные пути их решения.

---

## 1. Архитектура и структура кода

### Проблема: Нарушение принципа единственной ответственности (SRP) в PageModels
Файл `Pages/Game.cshtml.cs` выполняет слишком много функций:
- Обработка начальной загрузки данных (bootstrap).
- Роутинг AJAX-запросов для получения частичных представлений (Partial Views).
- Выполнение бизнес-логики (расчет наград мини-игры через сервис).
- Ручной маппинг сущностей базы данных в DTO.
- Логирование телеметрии.

**Решение:**
- Вынести обработку AJAX-запросов (телеметрия, расчет наград, каталог) в отдельные **API-контроллеры** (`Controllers/GameApiController.cs`).
- Использовать **AutoMapper** или вынести логику маппинга в слой бизнес-логики (Services).
- Разделить логику управления состоянием игры и логику отображения.

### Проблема: Сложная SQL-логика в репозиториях
В `PlanetRepository.cs` используются тяжелые SQL-запросы с ручным формированием JSON-массивов (`STRING_AGG`, `STRING_ESCAPE`). Это затрудняет поддержку при изменении схемы базы данных.

**Решение:**
- Рассмотреть использование **Entity Framework Core** для управления сложными связями (многие-ко-многим).
- Если используется Dapper, разбить запрос на несколько простых или использовать встроенные механизмы маппинга связей (Multi-mapping).
- Вынести JSON-десериализацию из репозитория в модели или сервисы.

### Проблема: Смешивание логики UI и состояния во фронтенде
В `stateManager.ts` присутствуют прямые манипуляции с DOM (`document.getElementById`, `classList.toggle`), что делает код жестко связанным с конкретной HTML-разметкой и затрудняет тестирование.

**Решение:**
- Использовать паттерн **Observer** или **Pub/Sub** для уведомления UI-компонентов об изменении состояния.
- Отделить "чистое" состояние (Store) от логики его отображения.

## 2. Качество кода и поддержка (Maintainability)

### Проблема: Магические строки и гигантские switch-конструкции
В `Game.cshtml.cs` (метод `OnGetPartial`) и `main.ts` используются большие блоки `switch` для определения экранов и действий. Это затрудняет добавление новых функциональных возможностей.

**Решение:**
- Использовать словарь (Dictionary/Map) или регистрацию через DI для динамического определения обработчиков экранов.
- Вынести константы (ID экранов, типы действий) в отдельные перечисления (Enum) или константные классы.

### Проблема: Неконсистентная работа с кэшированием
Логика кэширования реализована непосредственно внутри `PlanetRepository`. Это нарушает чистоту слоя доступа к данным.

**Решение:**
- Использовать паттерн **Decorator** для кэширования репозиториев.
- Создать `CachedPlanetRepository`, который оборачивает основной репозиторий и добавляет логику кэширования, не загрязняя основной код доступа к данным.

## 3. Безопасность

### Проблема: Глобальное отключение Antiforgery Token
Атрибут `[IgnoreAntiforgeryToken]` в `GameModel` отключает защиту от CSRF-атак для всех методов страницы.

**Решение:**
- Включить проверку токенов и настроить передачу `XSRF-TOKEN` в заголовках AJAX-запросов на фронтенде.
- Отключать защиту только для специфических эндпоинтов, если это абсолютно необходимо (например, для внешних вебхуков).

### Проблема: Прямая работа с innerHTML
В TypeScript-коде часто используется `innerHTML`, что потенциально открывает возможность для XSS-атак при отображении данных, полученных от пользователя или из базы.

**Решение:**
- Использовать `textContent` для текстовых данных.
- Для динамической генерации HTML использовать шаблоны (`<template>`) или манипуляции с DOM-узлами (`createElement`).

## 4. Организация проекта

### Проблема: Некорректное расположение исходных файлов TypeScript
Исходные файлы `.ts` находятся в директории `wwwroot/ts`. По стандартам веб-разработки, `wwwroot` должна содержать только готовые к раздаче статические файлы (скомпилированный JS, CSS, изображения).

**Решение:**
- Перенести папку `ts` в корень проекта (например, в `/Scripts` или `/Frontend`).
- Настроить `outDir` в `tsconfig.json` на `wwwroot/js`.

### Проблема: Наличие "мусорных" файлов
Файлы `test.cshtml` и `Test.cshtml.cs` выглядят как временные инструменты для отладки.

**Решение:**
- Удалить неиспользуемые тестовые файлы или перенести их в отдельный проект для тестов.

## 5. Анализ страницы настроек (Settings Page)

### Проблема: Полное отсутствие персистентности (Сохранения)
В коде присутствует функция `saveAndApply()` с выводом `alert('Настройки сохранены.')`, однако **настройки нигде не сохраняются**. Экшен `SET_SETTINGS` обновляет только in-memory Store (`stateManager.ts`). В отличие от состояния игрока (`_PLAYER_ACTIONS`), настройки не сериализуются в `localStorage` и полностью теряются при обновлении страницы.
Кроме того, при старте приложения не происходит загрузка и применение ранее выбранных настроек (например, громкости звука или масштаба UI).

**Решение:**
- Реализовать сохранение объекта `settings` в `localStorage` (например, под ключом `stellar_vocation_settings`).
- Добавить логику загрузки и применения настроек при старте приложения (bootstrap).

### Проблема: Применение настроек работает лишь частично или вхолостую
- Изменение ползунков громкости обновляет Store, но нигде не применяется к реальному аудио-движку (Three.js AudioListener или HTML5 Audio).
- `graphicsQuality` применяется дважды (при `onchange` селекта и при нажатии "Сохранить").
- `uiScale` применяется через изменение CSS-переменной `--user-ui-scale`, но из-за сброса при перезагрузке пользователю придется выставлять масштаб каждый раз.

**Решение:**
- Реализовать Service/Manager для управления звуком и музыкой, который будет реагировать на изменение `soundVolume` и `musicVolume`.
- Убрать дублирование вызовов (например, оставить мгновенное применение для предпросмотра, но фиксировать сохранение только при явном действии, либо сделать полное автосохранение без кнопки "Сохранить").

### Проблема: Использование глобальных переменных и inline-обработчиков
В файле `Pages/Game/_ScreenSettings.cshtml` логика привязана к глобальному объекту `window._settings` через inline-обработчики событий (`oninput`, `onchange`, `onclick`). В `wwwroot/ts/game/screens/screenSettings.ts` этот объект принудительно добавляется в `window`. Это нарушает инкапсуляцию, усложняет тестирование и создает риск конфликтов имен.

**Решение:**
- Удалить inline-обработчики из HTML.
- Назначать слушатели событий (Event Listeners) динамически внутри TypeScript файла (`screenSettings.ts`) при инициализации экрана (`init`), используя идентификаторы элементов.

### Проблема: Прямые манипуляции DOM и отсутствие реактивности
В `screenSettings.ts` используются прямые вызовы `document.getElementById` (`_setVal`, `_setChk`) для ручной синхронизации состояния Store с UI.

**Решение:**
- Использовать существующую систему подписок (`stateManager`) для автоматического обновления UI при изменении настроек.
- Реализовать однонаправленный поток данных: изменение в UI отправляет Action (через `dispatch`), а UI обновляется только по факту изменения в Store.

### Проблема: Злоупотребление inline-стилями
В `_ScreenSettings.cshtml` широко используются inline-стили (например, `style="width:100%;min-height:100vh;display:flex;..."`). Это раздувает разметку и усложняет поддержку тем (например, dark/light mode) или адаптивного дизайна.

**Решение:**
- Вынести стили в общие CSS-классы в отдельный файл (например, `wwwroot/css/game/settings.css` или общий файл стилей UI).
- Использовать уже имеющуюся систему utility-классов (если она есть) для стандартных отступов и флексбоксов.

## Заключение
Проект имеет хорошую базу и разделение на слои (Business/Data), однако текущая реализация SPA-хоста на Razor Pages начинает перегружаться. Фронтенд-часть требует избавления от антипаттернов (прямая манипуляция DOM, глобальные объекты), а также перехода к реактивному подходу. Рекомендуется провести рефакторинг в сторону более чистого разделения Frontend и API, стандартизировать работу с состоянием и стилями.
